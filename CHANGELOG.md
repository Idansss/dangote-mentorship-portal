# Changelog

## M0 — Foundations

- Next.js (App Router) + TypeScript strict, Tailwind CSS, shadcn-style UI primitives.
- Full Prisma/PostgreSQL schema for §5 (every table soft-deletable + cohort-scoped + bilingual-aware).
- Auth.js (NextAuth v5): Microsoft Entra ID SSO primary, email/password fallback, hashed invite links.
- Server-side RBAC for the six roles via `requireRole()`; edge route gating in `middleware.ts`.
- Cohort + Programme CRUD (Super-Admin-only) with Zod validation and audit logging.
- next-intl EN/FR scaffolding (cookie-based locale, no URL routing) with locale switcher.
- Seed: bilingual demo cohort — 15 mentors, 30 mentees, staff accounts, competency taxonomy, and a messy mentor import for the M1 validator.
- GitHub Actions CI: typecheck + lint + test; smoke/unit tests for roles, invites, and i18n parity.
- Invite creation: admin Invites page (create + copy-once link, list, revoke), Super-Admin-only for admin roles, audited, EN/FR, seeded demo invite, schema unit tests.
- Cohort & programme edit/archive UI wired to the existing update/archive actions (Super-Admin-only, confirm-before-archive).
- Login page hides the Microsoft Entra SSO button until real tenant credentials are configured (`isEntraConfigured()`).
- Playwright E2E (`npm run test:e2e`, port 3001): super-admin login → admin dashboard; unauthenticated redirect to login.
- Repo initialized as git; Husky pre-commit runs typecheck + lint + unit tests.

### M0 completion

- Entra SSO now links to admin-pre-created accounts of the same email (`allowDangerousEmailAccountLinking`) so SSO + admin-created accounts are one identity, not duplicates.
- Auth-endpoint rate limiting (§14): fixed-window limiter throttles credential sign-in (5/min per IP+email) and invite acceptance (10/min per IP); swappable store, unit-tested, with EN/FR "too many attempts" messaging.
- CI now runs the happy-path Playwright E2E against a Postgres service (migrate + seed + build + run). It serves a production build via `next start` instead of `next dev` to avoid first-request compile timeouts; verified locally via the same `CI=1` path (2/2 passing).
- Localized the root loading fallback (`app/loading.tsx`) — was hardcoded English.

### M0 deferred items (password reset, rate-limit store, lint)

- **Schema change:** added `password_reset_tokens` table (migration `20260610075205_password_reset_tokens`) storing only a hashed, single-use, 60-minute token.
- Password reset for credential accounts: public `/forgot-password` (no account enumeration — always reports "sent") → emailed link → `/reset-password/[token]` sets a new password, burns the token, signs the user in. Rate-limited, audited, EN/FR, with a "Forgot password?" link on login.
- Provider-agnostic mail transport (`lib/mail`, mirrors `lib/ai`): default "log" transport keeps the flow working before Resend/Graph is wired; gated by `MAIL_DEBUG` for body output.
- Rate limiter now sits behind a `RateLimitStore` seam (in-memory default, `setRateLimitStore()` to swap for Redis/Upstash) instead of a hardcoded map.
- Migrated `next lint` → ESLint flat config (`eslint.config.mjs`, `eslint .`); removed `.eslintrc.json`. No more Next 16 deprecation notice.
- Note: email verification has no entry point (accounts are seed- or invite-created, both pre-verified; no public self-registration), so no verification flow was built.

## Experience Layer (§18 Tier 1)

### Phase 0 — schema + cross-cutting plumbing

- **Schema change** (migration `20260610083433_experience_layer_tier1`): new tables `support_requests` (§1.13), `reflection_journal_entries` (§1.16), `mentor_private_notes` (§1.16), `goal_evidence` (§1.7), `form_drafts` (§1.11); new columns `goals.stage` + `goal_reviews.stage` (`GoalStage` enum, §1.7) and `meetings.did_happen`/`no_show_reason`/`no_show_reported_by(_at)` (`NoShowReason` enum, §1.14). Capture lands before anything computed from it. Confidentiality posture for reflections/private notes mirrors DMs (content private; translate toggles persist via the `translations` cache, not as columns).
- **Translation cache + AI adapter wired (first real consumer).** `lib/translation` exposes `translateContent()` backed by the `translations` table; pure decision core in `lib/translation/cache.ts` is unit-tested (golden case: identical content is translated once, then served from cache; degrades to original text when AI is unconfigured). A reusable `<TranslateToggle>` client component + `translateText` server action give every bilingual surface an opt-in EN↔FR toggle (CLAUDE.md §16).
- **Autosave drafts seam (§1.11).** `form_drafts`-backed `saveDraft`/`clearDraft` actions, server-side `getDraft()` loader, and a debounced `useFormDraft` hook for "never lose work". Drafts are transient (not audited; hard-deleted on submit); each later form adopts the hook as it is built.
- **Microsoft Graph mail transport (§2, 0c).** `lib/mail/graph.ts` sends via Graph app-only auth (cached token) behind the existing `getMailTransport()` seam; `isGraphMailConfigured()` gates it, falling back to the log transport when the `MAIL_GRAPH_*` vars are unset. New env vars documented in `.env.example`.

### Phase 1 — mentorship loop (M2 data capture)

- **Object storage seam (`lib/storage`).** Provider-agnostic `StorageProvider` (mirrors `lib/ai`/`lib/mail`) with a local-filesystem default that stores files outside `/public` under `STORAGE_LOCAL_DIR`; a Supabase Storage provider slots in behind the same interface later. Files are served only through authorized routes, never public URLs (§14). Added `pdf-lib` dependency.
- **Agreements e-sign + PDF (M2).** A matched pair can sign the **Mentoring** and **Confidentiality** agreements at `/agreements`: bilingual templates (EN/FR, versioned), typed-name + consent-checkbox signature, exact wording snapshotted into `terms_json`, and a server-rendered PDF (pdf-lib) stored via the storage seam. PDFs download through `/api/agreements/[id]/pdf` with server-side authz (signer or admin only — outside the auth middleware matcher, so enforced in-handler). Only participants in an `ACCEPTED` pairing may sign; one signature per type per cohort; every signature audited. Pure status helper (`summarizeAgreements`) + template completeness are unit-tested.
- **Schema change** (migration `20260610104345_meeting_pair_and_external_id`): `Meeting` gains `mentor_id`/`mentee_id` (the pairing it's for, mirroring `SessionLog`; `organizer_id` stays as whoever scheduled it) + `external_id` (the calendar event id), with indexes on both participants.
- **Meeting scheduler + `MeetingProvider` abstraction + no-show capture (M2, experience-layer.md §1.12/§1.14).** `/meetings` lets either party in an accepted pair schedule a session (title, type, start/end, location, notes); meetings list as **Needs-confirmation / Upcoming / Past**. The new `lib/meetings` provider seam (mirrors `lib/ai`/`lib/mail`/`lib/storage`): a no-op default keeps scheduling working dark, and an **Outlook (Microsoft Graph) provider** pushes the event to the organizer's calendar with the counterpart as attendee when Graph is configured (§1.12 write-path; "if it isn't in Outlook it doesn't exist") — best-effort, so a calendar failure never loses the meeting. Graph app-only auth was extracted to a shared `lib/graph/client` (mail + calendar reuse one cached token). **No-show capture (§1.14):** once a scheduled meeting's time passes with no recorded outcome, both participants get a one-tap "Did this meeting happen?" — "no" records a reason (mentor/mentee cancelled, forgot, rescheduled, other), feeding the risk monitor/heatmap with the *why*. Zoom join-link provisioning stays an M5 concern behind the same interface. Pure timing/resolution logic (`needsNoShowCapture`, `resolveNoShowReport`, `isValidWindow`) is unit-tested; every mutation is Zod-validated, pairing-authorized, and audited; EN/FR throughout. **Seed:** the demo pair gets one upcoming and one past (awaiting-confirmation) meeting.
- **Schema change** (migration `20260610102126_action_item_blocked_status`): added `BLOCKED` to the `ActionItemStatus` enum for the Action Item Tracker's four-state model (experience-layer.md §1.6).
- **Session logging + AI Session Assistant + Action Item Tracker (M2, experience-layer.md §1.4/§1.6/§1.15).** `/sessions` lets a mentor log a session against a paired mentee with the full §6.9 field set, or paste **rough notes** and have the **Session Assistant** structure them — clean summary, competency, extracted action items (task/owner/due), next steps, timeline, challenges, suggested next agenda, and a risk flag — all landing **editable** before the human saves (advisory only; degrades to a manual form when AI is off — CLAUDE.md §0 rule 5). The pure prompt builder + defensive parser live in `assistant.ts` and are unit-tested. **Action items** are first-class: created with the log (AI-extracted then edited) or added later, assigned to mentor/mentee, with a status control (Not started / In progress / Completed / Blocked) editable by the assignee or the pair's mentor, and overdue items flagged. Mentees add a **reflection** per log (§4: mentees reflect on their own); mentor notes stay mentor-only. The create form is **offline-resilient** (§1.15) via a new `useOfflineForm` hook — values mirror to localStorage instantly and a failed/offline submit is queued and retried automatically when connectivity returns, with a "saved locally / synced" indicator. Accepted-pair resolution was extracted to a shared `lib/pairings.ts` (now used by goals + sessions). Every mutation is Zod-validated, pairing-authorized, and audited; EN/FR throughout. **Seed:** the demo pair gets a sample session log with two action items.
- **Smart notifications rail — in-app + email + daily digest (experience-layer.md §1.10).** **Schema change** (migration `20260610113422_notifications_email_and_prefs`): `Notification` gains `email_pending`/`emailed_at` for the email channel; new `notification_preferences` table (email on/off, daily-digest on/off, per-type `muted_types[]`). A provider-agnostic emit seam (`lib/notifications`): `notify()` / `notifyMany()` render each message in the **recipient's** language (via `createTranslator`, not the actor's locale), write the in-app row, and route email per a pure, unit-tested rule (`selectEmailChannel`): in-app is always immediate; email goes **immediately for time-critical types** (meeting reminder, clinic-tomorrow, session-log-due, review-due) and is otherwise **batched into a daily digest** (`sendDailyDigests()`, ready for a scheduled job in M5). Muted types produce nothing; `notify()` never throws so a notification failure can't roll back its triggering action. Emitters wired into existing events: **goal reviewed → mentee**, **meeting scheduled → counterpart**, **support request → admins**, **support response → requester**. In-app centre at `/notifications` (deep-linked items, mark-read / mark-all-read, channel + per-type preferences) with an unread badge in the header. The remaining §1.10 types (profile-incomplete, match-ready, meeting-reminder, session-log-due, review-due, clinic-tomorrow) are defined and localized but emit when their date/cron-driven triggers land (M3/M4/M5). WhatsApp/SMS/Teams stay behind the existing feature flag (Tier 2). EN/FR throughout. **Seed:** the demo mentee has one unread + one read notification.
- **Anonymous support request + admin queue (experience-layer.md §1.13).** Any participant can "request support privately" at `/support`: a reason (can't reach partner / goal help / uncomfortable with match / need admin intervention / communication issue / language support / other) plus an optional message. The form states plainly that the request is **anonymous to other participants but visible to the programme team** (never to the partner) — a §1.13 UI requirement. Requests route to a private admin queue at `/admin/support` (admin-gated by route layout and re-checked in the action) where admins respond and move the request OPEN → IN_PROGRESS → RESOLVED; the response shows back to the requester on their own page. Requester cohort is resolved from their mentee/mentor profile, falling back to the active cohort so non-profile roles can still ask. Every mutation is Zod-validated and audited (audit carries reason/status only, never the message body); EN/FR throughout, with a pure unit-tested `countOpen` queue helper kept free of `server-only` so it tests cleanly. **Seed:** the demo mentee has one open request waiting in the queue.
- **Reflection journal (mentee) + private notes (mentor) (experience-layer.md §1.16).** `/journal` gives mentees a private reflection journal — free-text entries with post-session prompts (what I learned / action / challenge / feedback / support), an optional link to one of their own session logs, and a draft that autosaves via the §1.11 hook so nothing is lost. Entries are **private by default**; the mentee can **explicitly share** an individual entry with their mentor (an audited consent action) and unshare it again. Mentors get a separate **private notes** space per paired mentee (observation / strength / growth area / follow-up / next-session idea), never visible to the mentee or to admins. Both surfaces share the **DM confidentiality posture** (CLAUDE.md §7, §16): bodies stay private, audit rows carry ids/flags only — never content — and every body gets the §"bilingual everywhere" translate toggle. The safety-critical sharing rule is a pure, unit-tested `visibility.ts` (golden cases: an unshared entry, a non-mentee's entry, and a deleted entry are *never* visible to a mentor), re-applied as defense-in-depth on top of the query filter. Accepted-pair cohort resolution was promoted to a shared `mentorPairCohort()` in `lib/pairings.ts`. Every mutation is Zod-validated, RBAC- and ownership/pairing-authorized, soft-deletes, and is audited; EN/FR throughout. **Seed:** the demo mentee gets one shared and one private reflection; the demo mentor gets one private note.
- **Goals + AI Goal Coach + mentor approval + progress visualization (M2, experience-layer.md §1.7).** `/goals` gives mentees a SMART goal form (autosaving a draft via the §1.11 hook so work is never lost) and the **Goal Coach** — a deterministic SMART-readiness check plus, when the AI adapter is configured, an editable rewrite suggestion (advisory only; degrades gracefully when AI is off — CLAUDE.md §0 rule 5). Mentees submit; the paired mentor approves / requests changes / comments (approval is the human action that flips status — §7); approved goals advance through the **Drafted → Mentor reviewed → Approved → In progress → Evidence submitted → Achieved** stage timeline with a progress bar, and mentees upload **evidence** through the storage seam, served only via the authorized `/api/goals/evidence/[id]` route (uploader / paired mentor / admin). `GoalStatus` (approval workflow) and `GoalStage` (progress) are kept coherent by a pure, unit-tested `stage.ts` (review + mentee-advance transitions) and `smart.ts` (assessment, coach prompt builder, defensive response parser); every mutation is Zod-validated, RBAC- and pairing-authorized, and audited. EN/FR throughout. Added a `<Textarea>` UI primitive. **Seed:** one `ACCEPTED` demo pair (also unblocks the agreements demo) with two sample goals — one awaiting review, one approved and in progress.

### Phase 2 — computed experience surfaces

- **AI meeting preparation (experience-layer.md §1.5).** **Schema change** (migration `20260610115836_meeting_prep_cache`): `Meeting` gains `prep_json`/`prep_generated_at` to cache the prep per meeting. Both participants get a **Prepare** view (`/meetings/[id]/prepare`, linked from each upcoming meeting) that always shows the **real context** — previous session summary, pending action items, goals to review — assembled in one batched query, then an AI **suggested agenda / questions to ask / challenges / recommended resources**, generated **on demand** and cached on the meeting (with a Regenerate button). Degrades to a static prep built from the pending items + goals + a **Mentor-Guide question bank** when AI is off (CLAUDE.md §0 rule 5). The prompt builder, defensive JSON parser, and fallback are a pure, unit-tested `prepare.ts` (8 cases). Participant-only authz; generation is audited; EN/FR throughout. **Seed:** the demo pair's existing upcoming meeting is immediately preparable.
- **Mentorship Journey Tracker (experience-layer.md §1.2).** A persistent roadmap of the 9-month journey (Profile → Training → Matched → Confidentiality → Goals → Monthly sessions → Mid-term → Final → Completion), shown on the mentee and mentor dashboards. Each step is one of **Completed / Your turn / Overdue / Upcoming**, **computed from real records** (profile completeness, training status, accepted match + age, signed confidentiality agreement, goal statuses, session count/recency) — never manually set, with deep links to the relevant page and an overall progress %. The mapping is a pure, unit-tested `journey.ts` (17 cases: cross-step gating, mentee-vs-mentor goals semantics, time-based overdue thresholds, 0→100% progress); the data layer gathers facts in one batched query per role. Matching stays `Upcoming` (admin-driven, never the participant's action). Mid-term/final/completion remain `Upcoming` until M3 models per-user review state — defined now, enriched then. Mobile-first card with status colours + progress bar; EN/FR. No schema change (derived `journey_state`, per §1.2).

### Phase 3 — shared surfaces, guidance & navigation

- **Pair Contract Page (experience-layer.md §1.8).** `/pair` is the matched pair's shared home base: one read-only view aggregating the records they already share — both **agreements** (per-party signed status), **meetings** (next meeting + held/logged counts + last session), **goals** (stage + status), and **open action items** (assignee, due/overdue) — each section deep-linking to where the work is actually done. A mentee lands straight on their own pair; a mentor picks from their mentees. Access is the safety-critical bit: a pure, unit-tested `pairAccessFromMatch` (6 cases) resolves the viewer's role from the accepted `Match` and is re-applied on top of the scoped query, so only the two people in the accepted pair can ever open it (admins are not viewers here — it is the pair's private space). Still-`DRAFT` goals stay the mentee's alone. The **direct-messaging entry point** is shown but lands with the M4 community features (flagged, not stubbed). No schema change. Mobile-first cards; EN/FR; linked from the header nav.
- **Onboarding tour + Help Center (experience-layer.md §1.11).** Completes §1.11 (autosave drafts already shipped). **First-login guided tour** per role: a dismissible, step-by-step card on the mentee and mentor dashboards (5 steps each, bilingual), shown once and never again unless re-requested — dismissal persists in `localStorage` (no schema change), and a "Replay the tour" control in the Help Center re-arms it. **Help Center** at `/help`: short, searchable, bilingual articles (how matching works, submitting goals, scheduling, session logs, journal, translation, requesting support), content held statically in the i18n catalogue with a typed slug registry; client-side instant search, article pages render paragraph bodies. Clinics/reviews articles join when those features exist, so Help never documents a screen that isn't there. Linked from the header nav; EN/FR.
- **Calendar views (experience-layer.md §1.12).** `/calendar` adds **month / week / list** read views over the user's meetings plus **programme milestones** (cohort start/end), completing §1.12 alongside the already-shipped Outlook write-path. Navigation (view switch, prev/next/today) is **entirely URL-param driven**, so the page stays server-rendered and fast (performance budget). All date maths — Monday-based weeks, the full-week month grid, view ranges, stepping, day-bucketing — is a pure, unit-tested `calendar.ts` (13 cases) handled in UTC. The event model is open for clinics (M4), review deadlines (M3), and training sessions to slot in as those features land; admins see active-cohort milestones. Month grid scrolls horizontally on small screens; EN/FR month and weekday names; linked from the header nav.
- **Quick Actions menu (experience-layer.md §1.9).** A floating "+" (large touch target, fixed bottom-right) in the authenticated participant layout opens the shortest path to the common flows — schedule meeting, add session log, submit goal, add reflection, request support. Items are **role-filtered server-side** (a mentee sees "submit goal", a mentor "add session log") and passed to a small client menu that closes on outside-click/Escape and is keyboard- and `aria`-labelled. "Join clinic" joins when the M4 community features land. EN/FR.
- **Personalized role dashboards (experience-layer.md §1.1).** The M0 stub dashboards become "what matters to me right now," above the fold. **Mentee:** my-mentor card (name, role, language) linking to the pair workspace, next meeting, current goals with per-goal **progress bars** (reusing the §1.7 `stageProgressPercent`), pending action items (due/overdue), the §1.3 button, the journey tracker, and a rotating bilingual **weekly tip** (ISO-week indexed, no storage). **Mentor:** my-mentees cards (each linking to `/pair/[menteeId]`), pending-goal-reviews count, next meetings, and **session logs awaiting completion** (meetings that happened with no log yet). **Admin:** programme-at-a-glance tiles — active pairs, unmatched mentees/mentors, open support requests, goals submitted/approved, upcoming meetings, training completed — each deep-linked. All counts come from a batched read model in `features/dashboard/data.ts`; every figure is a real record. Cards needing later milestones (upcoming clinic + clinic attendance M4, "mentees who need attention" + review deadlines + at-risk pairs M3, programme health score Tier 2) are **absent, not stubbed** (CLAUDE.md §16). Mobile-first; EN/FR.
- **"What should I do next?" (experience-layer.md §1.3).** A single button on the mentee and mentor dashboards (admin lands with §1.1). On tap, `getNextBestAction()` assembles the user's **live, deep-linked candidates** from real records — overdue action items, meetings awaiting confirmation, no upcoming meeting, goal changes requested, goals to set, pending goal reviews (mentor), open support requests + matches awaiting approval (admin) — and asks the AI to choose and phrase **one** prioritized action. **Grounding is enforced:** the candidate links are the only allowed outputs, and a pure, unit-tested `parseNextActionResponse` (8 cases) **rejects any link the model invents**, falling back to the highest-priority candidate; with AI off it returns that deterministic top pick. Result is **cached 10 minutes** per user+language (in-process, mirrors the rate-limiter store). Read-only — it suggests, never writes. Messages are localized in the user's language so the model receives readable, grounded facts. EN/FR throughout.
- **First-session icebreaker (experience-layer.md §1.17).** **Schema change** (migration `20260610120500_match_icebreaker_cache`): `Match` gains `icebreaker_json`/`icebreaker_generated_at` to cache the per-pair guide. When a pair is active, both participants see a personalized first-meeting guide built from **both profiles** — shared interests, opening questions, what the mentee wants to learn, what the mentor offers, and a light agenda — generated **on demand** and cached on the match (Regenerate button). Always shows a guide: degrades to a **profile-built fallback** when AI is off (CLAUDE.md §0 rule 5), including a case-insensitive shared-interest match. The interest match, prompt builder, defensive parser, and fallback are a pure, unit-tested `icebreaker.ts` (10 cases). Attached to the **Pair Contract Page** and to the **first meeting's Prepare view** (shown only when no session is logged yet). Participant-only authz (reuses `pairAccessFromMatch`); generation is audited; EN/FR throughout. Grounded strictly in the two profiles — no fabricated facts.

### Step 3 — signature components (§5–§7)

- **Journey Rail (§5).** The signature element: a 9-step roadmap (Profile → … → Certificate), horizontal on desktop / vertical on mobile, on a connecting line that **fills green up to the current step** (load-once animation, reduced-motion safe). Node appearance is driven by the computed journey state (experience-layer.md §1.2) — completed (filled + check), current (green ring + pulse / heavier ring under reduced motion), needs-action & overdue (amber/red ring + dot badge), upcoming (hollow). Every node deep-links and shows a tooltip (step · status · the one action). Built as a server wrapper (`journey-rail.tsx`, translates labels) + a client view (`journey-rail-view.tsx`, tooltips + animation); **drop-in replacement** for the old `JourneyTracker` (same `{ result }` prop) — the mentee & mentor dashboards now render the rail full-width, and the off-spec tracker (raw `bg-green-600`/`bg-amber-500` etc.) was removed. New i18n key `journey.open` (EN/FR). All colour from tokens.
- **Bilingual field control (§6).** `BilingualField` — the compose control (label + input/textarea + a 🌐 EN|FR chip emphasizing the writer's active language) so French users write French (CLAUDE.md §16). `BilingualContent` — the read control: saved content with a quiet source-language tag and the existing on-demand, cached `TranslateToggle`. Adoption across the feature forms happens in step 4.
- **AI output container (§7).** `AIContainer` — one consistent visual language for AI as a native citizen: an `--info` blue spark, a light blue-tinted container, a title, an "editable" hint, and an actions row — so an AI suggestion always reads as an editable draft distinct from committed data. Plus `AISpark` for compact inline markers. Adoption across the AI surfaces (session summary, meeting prep, goal coach, "what next?") happens in step 4.
- All three are showcased in the `/design` preview.

### Phase 4 — scheduled notification triggers (completes §1.10)

- **Date/cron-driven notification emitters + scheduled daily digest (experience-layer.md §1.10).** Closes the last independently-buildable §1.10 gap: the notification infrastructure (in-app + immediate/digest email + per-user prefs) already shipped, but the date-driven types never fired and the digest had no caller. A secret-protected route handler `/api/cron/notifications` (auth via `Authorization: Bearer <CRON_SECRET>`, **fails closed** with 503 when unset; portable to Vercel Cron or an on-prem Task Scheduler/systemd timer) runs `runScheduledNotifications()`, which emits the types whose data exists today and then sweeps `sendDailyDigests()`. Emitters wired: **meeting_reminder** (meetings starting within 24h, to both participants, date localized to each recipient's locale + timezone), **session_log_due** (past meetings beyond a 2h grace with no log and not a confirmed no-show, to the mentor), **profile_incomplete** (incomplete mentor/mentee profiles in active cohorts). **match_ready** is wired event-driven instead (on admin approve/override, to both participants). Every emitter is **idempotent**: meeting/session nudges dedupe per-meeting on `(type, link)` (the meeting id rides in the link fragment), profile nudges dedupe on a 7-day window, so a missed or duplicated run never spams. The pure timing/completeness predicates (`needsMeetingReminder`, `needsSessionLogReminder`, `isMentorProfileIncomplete`, `isMenteeProfileIncomplete`) live in `lib/notifications/schedule.ts` and are unit-tested (14 cases incl. window edges, grace, confirmed-no-show exclusion, zero-vs-null experience). New env var `CRON_SECRET` documented in `.env.example`. **review_due** (M3 reviews) and **clinic_tomorrow** (M4 clinics) stay deferred until those features capture their data (CLAUDE.md §16). EN/FR throughout (recipient's language, via the existing emit seam).
